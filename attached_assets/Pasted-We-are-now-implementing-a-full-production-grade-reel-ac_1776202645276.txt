# CRITICAL RULE 2 — MENU MUST CHANGE BASED ON USER ROLE

We have TWO completely different menu states:

## CASE A — VIEWER (user is NOT the owner of the auction)

Menu MUST contain ONLY:

1. Report
2. Interested
3. Not Interested
4. Mention users

NOT allowed here:

* Delete
* Owner actions
* Duplicate share

---

## CASE B — OWNER (user IS the creator of the auction)

Menu MUST contain ONLY:

1. Delete
2. Share (optional here ONLY if we decide owner needs it — otherwise rely on main share button)

NOT allowed:

* Report (cannot report your own content)
* Interested / Not Interested (nonsense on own content)
* Mention (optional — only if product decision allows)

Also:

* REMOVE "Download" completely (not needed for own content)

# CRITICAL RULE 4 — INTERESTED / NOT INTERESTED (REAL SIGNALS)

These are NOT buttons — they are feed signals.

## Requirements:

### Interested:

* Send signal → content_signals table (already exists)
* Mark visually active
* Remove "not interested" if previously selected

### Not Interested:

* Send negative signal
* Immediately reduce visibility in feed (existing system supports this)
* Mark visually active

### UX:

* Only ONE state active at a time
* Toggle behavior must be correct
* No duplicate requests

---

# CRITICAL RULE 5 — MENTION SYSTEM (STRICT RULE)

User requested HARD constraint:

### Mention ONLY allowed when:

* I follow the user
* AND the user follows me

This is a mutual-follow relationship.

## Required behavior:

### When opening "Mention":

* Show ONLY users where:
  mutual_follow = true

### NOT allowed:

* One-way follow
* Random users
* Full user list

## UI:

* Search field
* List of eligible users only

## If no eligible users:

* Show empty state:
  "You can only mention users who follow you back"
