We are updating the BidReel production roadmap.

A new core product decision is now final:

## Official bidding logic

Bidders must NOT enter the final bid price.
They only enter the INCREMENT amount they want to add on top of the current price.

### Formula

new_price = current_price + bid_increment

### Rules

* bid_increment must be >= 1
* 0 is invalid
* negative values are invalid
* seller still sets the starting price when creating the auction
* every later bid is an increment only

This is now the official bidding model for the product.

Inspect the current bidding implementation across the full stack and identify:

1. how auction price is currently stored

3. whether frontend currently sends:

   * final price
   * absolute bid amount
   * or something else

5. what DB fields/tables are involved
6. what API routes / handlers are involved
7. what UI components / modals / forms are involved

## Then produce a safe refactor plan

Design the refactor in 4 sub-parts:

### Part 2.1 — Bid UX refactor

Frontend changes so the bidder sees:

* current price
* increment input
* preview of resulting price before submit

Text direction:

* do not say "enter your bid"
* say "How much do you want to increase?"
* show preview:

  * Current price: X
  * Increase: Y
  * New price: Z

### Part 2.2 — Bid business logic refactor

Refactor backend so:

* client submits increment amount only
* server calculates final new price
* server never trusts client-sent final price
* validation enforces increment >= 1

### Part 2.3 — Concurrency / integrity

Plan safe handling for simultaneous bids:

* avoid race conditions
* prevent incorrect price overwrites
* ensure bidding is atomic / consistent

### Part 2.4 — Realtime / refresh impact

Audit how this change affects:

* feed price updates
* auction detail page
* bid history
* notifications
* admin reporting
